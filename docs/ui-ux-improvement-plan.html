AprenDB UI/UX implementation plan Proposed PR sequence

Make AprenDB feel like a visual SQL tutor

This plan turns the current visual query sandbox into a clearer learning experience. The work is split into focused PRs so the team can ship user-visible value early while keeping review and testing manageable.

Product Goals

The app already has a strong foundation: schema graph, generated SQL, lesson goals, and result validation. These changes should make the next step obvious, explain failures, and reduce hidden interactions.

Teach while building

Replace generic status messages with live lesson progress and specific feedback tied to FROM, SELECT, JOIN, WHERE, ORDER BY, and LIMIT.

Reveal available actions

Make table and column actions discoverable without relying on trial-and-error clicks or context menus.

Fit real working screens

Keep the graph, lesson, SQL, and result areas usable on laptop and tablet-sized screens through responsive panels and stable layouts.

Recommended PR Breakdown

Build in this order. Each PR has a coherent product outcome and can be reviewed without requiring the later work.

PR Theme User outcome Primary files
1 Learning feedback Users know which lesson requirements are complete and why a result failed. src/App.tsx, src/lib/validation.ts, src/types/query.ts
2 Interaction discoverability Users can see what to click next and apply actions with less hidden menu work. src/App.tsx, src/data/schema.ts
3 Responsive layout The app remains usable below the current fixed desktop width. src/App.tsx, src/index.css
4 Workflow polish Users can move faster with copy SQL, next lesson, and keyboard shortcuts. src/App.tsx, focused component helpers as needed

Implementation Timeline

Each phase includes implementation tasks, acceptance criteria, and test notes. Keep commits small inside each PR so behavior changes are easy to inspect.

PR 1 Learning feedback
Add a lesson requirement model Must have

Represent lesson goals as explicit requirements instead of only prose tips. Start with a lightweight model derived from the current lesson data.

  • Add optional requirements metadata to lessons or create a helper that derives requirements from known lesson IDs.
  • Support requirement types: base table, selected columns, joined tables, filters, sort, limit, and expected result.
  • Return requirement state as complete, incomplete, or needs-preview.
Render live lesson checklist Must have

Replace or supplement the current static tips with a checklist that updates as the visual query changes.

  • Show completion state with text and an icon, not color alone.
  • Keep tips visible as secondary guidance for each requirement.
  • When no query exists, highlight the first recommended action.
Add actual vs expected result diagnostics Must have

Extend validation so failed previews return useful mismatch details instead of only passing or not passing.

  • Report wrong columns, wrong row count, missing rows, extra rows, and order differences.
  • Render a compact feedback panel above the result table.
  • Keep exact pass behavior unchanged for lessons that already pass.
Acceptance criteria
  • Each current lesson shows at least two live requirements.
  • Changing selected columns or joins updates the checklist immediately.
  • A wrong query preview explains at least one concrete mismatch.
  • Existing tests pass, and new tests cover validation diagnostics.
PR 2 Discoverability
Add canvas empty state Must have

When query.baseTableId is empty, show an unobtrusive overlay that guides the user to choose a starting table.

  • Use the active lesson to suggest the most likely starting table.
  • Provide one-click table start buttons.
  • Hide the overlay as soon as a base table is selected.
Make column actions visible Must have

Keep the action menu, but add inline affordances so users understand that columns can be selected, filtered, or sorted.

  • Add small hover/focus icons for add-to-result, filter, and sort actions.
  • Preserve the current click target for users who already discovered it.
  • Ensure controls are keyboard reachable.
Add filter value hints

Add optional sample values to schema columns, then show them in the filter UI. Example: orders.status can suggest paid, pending, and refunded.

Add relationship legend

Explain graph edge states: joined, available, and unavailable. Keep the legend compact and visually tied to the existing edge colors.

Acceptance criteria
  • A new user can start lesson 1 without opening a hidden menu first.
  • Column actions are visible on hover and reachable by keyboard.
  • Filter hints appear for columns with known sample values.
  • Graph relationship states are explained on screen.
PR 3 Layout
Remove fixed body minimum width Must have Layout risk

Replace body { min-width: 1024px; } with responsive rules. This is the main layout constraint currently preventing smaller screens from working.

Introduce collapsible or tabbed side panels

Preserve the three-pane desktop experience, then switch to a more compact mode on narrower screens.

  • Desktop: lessons left, graph center, SQL/result right.
  • Medium screens: collapsible lesson panel, graph center, right panel visible.
  • Small screens: tabs for Lessons, Graph, SQL, Results, and Query.
Stabilize result table and SQL panel
  • Add sticky result headers.
  • Ensure generated SQL wraps without pushing the panel wider.
  • Keep Preview Result button visible in the active query panel.
Acceptance criteria
  • The app is usable at 390px, 768px, 1024px, and 1440px viewport widths.
  • No text overlaps panel boundaries or buttons.
  • The graph remains pannable and zoomable in each layout mode.
  • Preview Result is always reachable without horizontal scrolling.
PR 4 Polish
Add Next lesson flow

After a lesson passes, show a clear next action. If the user is on the last lesson, show a completion state.

Add Copy SQL

Add a copy button to the generated SQL panel with copied/error feedback. Use the Clipboard API with a fallback message if unavailable.

Clarify query reset actions

Reset and New Query currently perform the same action. Keep one primary action or give them distinct meanings.

Add keyboard shortcuts
  • Escape closes menus and overlays.
  • Ctrl+Enter previews the result.
  • Ctrl+Shift+R resets the current query after confirmation.
Acceptance criteria
  • Passing a lesson offers a clear next step.
  • Generated SQL can be copied with visible feedback.
  • Duplicate header actions are resolved.
  • Keyboard shortcuts do not interfere with form input typing.

Component And File Map

This map shows where the current code is likely to change. Keep new helpers near the existing query and validation modules unless the file becomes difficult to scan.

src/App.tsx

Main layout, lesson sidebar, React Flow canvas, action menu, SQL panel, result panel, query summary, and preview behavior. Expect most UI component additions here at first.

src/data/lessons.ts

Add requirement metadata, suggested start tables, and lesson-specific guidance if the team chooses a data-driven checklist model.

src/data/schema.ts

Add optional column sample values, filter hints, or beginner-friendly descriptions for fields like orders.status and payments.method.

src/lib/validation.ts

Extend boolean validation into structured diagnostics while preserving the existing pass/fail function for compatibility.

src/lib/query-model.ts

Reuse existing helpers for joined tables, selected columns, filters, sort, and limit. Add small selectors only when checklist logic needs them.

src/index.css

Update global responsive constraints, theme colors if needed, and layout utility rules.

Suggested Data Shapes

These are starting points, not final API contracts. They are intentionally small so the first PR can land without a large architecture pass.

Lesson requirement

type LessonRequirement =
  | { kind: "base-table"; tableId: string; label: string }
  | { kind: "selected-columns"; columns: ColumnRef[]; label: string }
  | { kind: "join"; tableIds: string[]; label: string }
  | { kind: "filter"; column: ColumnRef; value?: string; label: string }
  | { kind: "sort"; column: ColumnRef; direction: "ASC" | "DESC"; label: string }
  | { kind: "limit"; value: number; label: string }
  | { kind: "result"; label: string };

Validation diagnostic

type ResultDiagnostic = {
  kind:
    | "wrong-columns"
    | "missing-rows"
    | "extra-rows"
    | "wrong-order"
    | "row-count";
  message: string;
  expected?: unknown;
  actual?: unknown;
};

Testing Plan

Use unit tests for query and validation behavior, then manually verify the visual flow in the browser because React Flow interactions are central to the experience.

Automated checks

  • Run pnpm test after each PR.
  • Run pnpm typecheck before review.
  • Add tests for structured result diagnostics in validation.test.ts.
  • Add tests for any checklist selector helpers if they live outside React components.

Manual checks

  • Complete every lesson from a fresh page load.
  • Intentionally make one wrong query per lesson and confirm feedback is useful.
  • Check mouse and keyboard access to table and column actions.
  • Verify layouts at mobile, tablet, laptop, and desktop widths.

Risks And Decisions

These are the choices most likely to affect scope. Resolve them before or during the first PR to prevent drift.

How data-driven should lessons be?

A requirement model makes future lessons easier, but deriving checklist rules from current lesson IDs is faster. Prefer data-driven requirements if more lessons are expected soon.

How strict should result order be?

Some SQL lessons should require ordering, while others should not. Validation diagnostics should distinguish wrong data from correct data in a different order.

Menu vs inline controls

Inline actions improve discoverability but can clutter table nodes. Start with visible hover/focus controls and keep the current menu as the complete action surface.

Responsive graph behavior

React Flow works on small screens, but node density and side panels may compete for space. Favor tabs or drawers on small screens instead of compressing all panels at once.

Definition Of Done

Use this final checklist before considering the UI/UX improvement track complete.

  • Users receive live feedback before previewing results.
  • Failed previews explain what is wrong in plain language.
  • First-time users can identify the next action without reading source code or guessing hidden gestures.
  • The app works without horizontal scrolling at common viewport widths.
  • Each shipped PR includes focused tests or a clear manual verification note.
  • Core app behavior remains intact: build query, generate SQL, execute preview, pass lessons.
